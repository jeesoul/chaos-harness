# PRD Validator

## 核心理念

**PRD 是研发需求的源头，源头质量决定了整个项目的成败。**

不要等到开发中途才发现需求不清晰。PRD 检查是预防性的质量门禁。

## 自动触发

| 触发条件 | 检测方式 | 动作 |
|---------|---------|------|
| 检测到 PRD 文件 | `*PRD*.md`, `*需求*.md` | 推荐 PRD 检查 |
| P02 阶段完成 | `P02-memory.yaml` 存在 | 自动检查 PRD 质量 |
| 用户说"检查 PRD" | 自然语言匹配 | 执行完整检查 |

## 检查项（基于铁律）

### IL-PRD001: PRD 必须包含验收标准

**检查规则**：
- PRD 中每个功能是否包含验收标准（`验收标准`、`Acceptance Criteria`）
- 验收标准是否可测试（避免"体验好"、"响应快"等模糊描述）
- 是否包含量化指标（数字、阈值）

**检测方法**：
```bash
# 读取 PRD 文件，检查验收标准
grep -c "验收标准\|Acceptance Criteria\|验收条件" output/*/docs/PRD.md
grep -c "作为.*我希望.*以便" output/*/docs/PRD.md  # 用户故事格式
```

### IL-PRD002: 用户故事必须可追溯

**检查规则**：
- 每个用户故事是否标注了需求来源
- 需求来源是否可追溯到原始记录（用户访谈、埋点数据、竞品分析）
- 功能是否关联到具体业务目标

### 结构完整性

| 检查项 | 必须存在 | 检测关键词 |
|--------|---------|-----------|
| 背景 | 是 | `背景`, `产品背景`, `业务目标` |
| 用户分析 | 是 | `用户画像`, `目标用户`, `用户场景` |
| 功能清单 | 是 | `功能清单`, `功能列表`, `功能名称` |
| 用户故事 | 是 | `作为.*我希望.*以便` |
| 验收标准 | 是 | `验收标准`, `Acceptance Criteria` |
| 非功能需求 | 是 | `性能`, `安全`, `兼容性` |
| 风险评估 | 推荐 | `风险`, `风险评估` |
| 里程碑 | 推荐 | `里程碑`, `排期`, `时间节点` |

## 执行流程

1. 定位 PRD 文件（`output/{version}/docs/PRD.md` 或用户指定路径）
2. 逐条检查上述检查项
3. 生成 PRD 质量报告
4. 标注通过/警告/失败项
5. 记录检查结果到 `learning-log.json`

## 输出格式

```markdown
# PRD 质量报告

**PRD**: output/v1/docs/PRD.md
**版本**: {version}
**检查时间**: {timestamp}

## 检查结果

| ID | 检查项 | 状态 | 说明 |
|----|--------|------|------|
| Q001 | 背景完整 | ✅ 通过 | 包含业务目标和背景描述 |
| Q002 | 用户分析 | ⚠️ 警告 | 用户画像存在但缺少用户场景 |
| Q003 | 功能清单 | ✅ 通过 | 包含 N 个功能，优先级标注完整 |
| Q004 | 用户故事 | ✅ 通过 | 所有功能均包含用户故事 |
| Q005 | 验收标准 | ❌ 失败 | F003/F005 缺少量化验收标准 |
| Q006 | 非功能需求 | ✅ 通过 | 包含性能和安全需求 |
| Q007 | 需求追溯 | ⚠️ 警告 | 30% 功能未标注需求来源 |

## 总结

- 通过: N 项
- 警告: N 项（需关注）
- 失败: N 项（必须修复）

## 建议

1. F003 验收标准需要量化（"体验好" → "页面加载 < 2s"）
2. 补充用户场景描述
3. 标注所有功能的需求来源
```

## 铁律

| ID | 铁律 | 说明 |
|----|------|------|
| IL-PRD001 | PRD 必须包含验收标准 | 每个功能都有可测试的验收标准 |
| IL-PRD002 | 用户故事必须可追溯 | 每个需求可追溯到来源 |
| IL-PRD003 | 技术方案变更需要记录原因 | P02 涉及技术方案时需记录 |

## 快捷命令

| 你说 | 动作 |
|------|------|
| "检查 PRD" | 执行完整 PRD 质量检查 |
