# Test Assistant

## 核心理念

**测试不是事后补充，而是质量的前置保障。**

好的测试工程师不只是写测试代码，而是：
1. 从需求出发推导测试场景（不是对着代码想测什么）
2. 自动化重复劳动，让精力集中在复杂场景
3. 用数据说话（覆盖率、通过率、回归对比）

## 自动触发

| 触发条件 | 检测方式 | 动作 |
|---------|---------|------|
| 检测到测试文件 | `*.test.*`, `*.spec.*` | 推荐覆盖率检查 |
| P08 阶段开始 | `P07-memory.yaml` 存在 | 推荐 E2E 测试生成 |
| 用户提到回归测试 | "回归", "对比", "之前版本" | 启动回归测试对比 |
| 用户提到 API 测试 | "API 测试", "接口验证" | 启动 API 自动化验证 |

## 能力模块

### 模块 1: 测试用例自动生成（P08）

**输入**：PRD 用户故事、功能清单

**能力**：
- 从用户故事自动推导 Given-When-Then 测试用例
- 覆盖正常流程 + 异常流程 + 边界条件
- 标注优先级（核心路径/重要路径/边缘场景）
- 输出 Playwright 或原生 JS 格式

**输出**：
```markdown
# 测试用例 - F001 用户登录

**来源**: PRD F001 - 用户故事 "作为用户我希望登录以便访问个人内容"

## TC-001: 正常登录（核心路径 P0）
- Given 用户已注册账号 test@example.com
- When 输入正确用户名和密码，点击登录
- Then 页面跳转到首页，显示欢迎消息
- 验证点: URL 包含 /home, 页面包含用户名

## TC-002: 密码错误（异常流程 P1）
- Given 用户已注册账号
- When 输入错误密码，点击登录
- Then 显示错误提示 "用户名或密码错误"，停留在登录页
- 验证点: 错误提示文本存在, 页面未跳转

## TC-003: 空字段提交（边界条件 P2）
- Given 用户在登录页
- When 用户名为空，点击登录
- Then 显示 "请输入用户名"
- When 密码为空，点击登录
- Then 显示 "请输入密码"

## TC-004: SQL 注入尝试（安全 P1）
- Given 用户在登录页
- When 用户名输入 "' OR 1=1 --"
- Then 登录失败，不暴露 SQL 错误信息
```

### 模块 2: E2E 测试脚本生成

**输入**：测试用例清单

**能力**：
- 生成 Playwright 格式的可执行 E2E 脚本
- 生成 web-access CDP 格式的直接浏览器操作脚本
- 自动配置等待策略和重试逻辑
- 输出到项目测试目录

**输出格式**：
```typescript
// Playwright 格式
import { test, expect } from '@playwright/test';

test('TC-001: 正常登录', async ({ page }) => {
  await page.goto('/login');
  await page.fill('[name="username"]', 'test@example.com');
  await page.fill('[name="password"]', 'password123');
  await page.click('button[type="submit"]');
  await expect(page).toHaveURL(/.*home/);
  await expect(page.locator('.welcome')).toContainText('Welcome');
});

// web-access CDP 格式（无 Playwright 项目回退）
// 通过 CDP eval 直接操作浏览器验证
```

### 模块 3: 测试覆盖率检查

**输入**：项目代码

**能力**：
- 运行测试并收集覆盖率报告
- 分析未覆盖的代码路径
- 基于 IL-TEST001 检查核心流程覆盖
- 生成覆盖率改进建议

**检查规则**：
| 级别 | 覆盖率要求 | 适用场景 |
|------|-----------|---------|
| 核心业务逻辑 | ≥ 90% | 支付、鉴权、数据变更 |
| 一般业务逻辑 | ≥ 80% | CRUD、查询、展示 |
| 工具类/辅助函数 | ≥ 60% | 格式化、验证器 |

### 模块 4: 回归测试报告对比

**输入**：当前测试报告 + 历史基准报告

**能力**：
- 对比测试通过率变化
- 对比覆盖率变化
- 标注新增/修复的失败用例
- 生成回归差异报告

**输出**：
```markdown
# 回归测试对比报告

**基准版本**: v1.2.0 (2026-04-01)
**当前版本**: v1.3.0 (2026-04-14)

## 总览
| 指标 | 基准 | 当前 | 变化 |
|------|------|------|------|
| 测试用例 | 120 | 135 | +15 |
| 通过 | 118 | 132 | +14 |
| 失败 | 2 | 3 | +1 |
| 覆盖率 | 82% | 85% | +3% |

## 新增失败用例
- TC-089: 用户列表分页异常（P08 新增）

## 已修复失败用例
- TC-023: 登录超时处理（已修复）

## 覆盖率变化
| 文件 | 基准 | 当前 | 变化 |
|------|------|------|------|
| src/login.ts | 78% | 92% | +14% |
| src/dashboard.ts | 65% | 62% | -3% ⚠️ |
```

### 模块 5: API 自动化验证

**输入**：API 文档或接口列表

**能力**：
- 从 API 文档自动推导验证用例
- 启动 dev server 后通过 CDP 验证接口响应
- 验证请求/响应格式、错误码、边界条件
- 生成 API 测试报告

**验证项**：
| 类型 | 验证内容 |
|------|---------|
| 正常响应 | 状态码、响应体结构、数据类型 |
| 参数校验 | 必填字段、格式校验、长度限制 |
| 错误处理 | 400/401/403/404/500 响应 |
| 性能基准 | 响应时间 < 阈值 |

## 铁律

| ID | 铁律 | 说明 |
|----|------|------|
| IL-TEST001 | E2E 测试必须覆盖核心流程 | 核心用户路径必须有自动化测试 |
| IL-TEST002 | 性能测试必须有基准 | 性能指标必须有可对比的基准值 |
| IL-TEST003 | 安全漏洞必须修复 | 发现的安全漏洞必须修复并重新测试 |
| IL003 | 完成需验证 | 测试通过必须有实际运行证据 |
