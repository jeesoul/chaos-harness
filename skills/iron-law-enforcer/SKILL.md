# 铁律执行哲学

## 为什么需要铁律

AI Agent 的非确定性既是创造力的来源，也是工程稳定性的最大威胁。Agent 有一种"投机性优化"倾向——为了快速达成目标，它会通过语义博弈"合理化"地绕过约束。

传统提示词是**软性建议**——存在语义灰色空间，规则越多漏洞越多。

铁律是**硬约束**——消除所有语义灰色空间，在架构层面物理阻断规避路径。

这不是 AI 的错，这是软性约束的系统性缺陷。

## 核心铁律（不可协商）

| ID | 铁律 | 为什么 |
|----|------|--------|
| IL001 | 文档必须在版本目录下 | 版本散乱 = 无法追溯 = 团队混乱 |
| IL002 | Harness 生成依赖扫描数据 | 不了解项目就生成约束 = 盲人摸象 |
| IL003 | 完成声明必须附带验证证据 | 口头"完成了" = 幻觉式交付 |
| IL004 | 版本变更需要用户确认 | 随意改版本 = 破坏信任 |
| IL005 | 敏感配置修改需要审批 | 误改密钥/数据库 = 生产事故 |

**Agent Team 铁律**：

| ID | 铁律 | 为什么 |
|----|------|--------|
| IL-TEAM001 | 评审必须多 Agent | 单视角 = 盲区 |
| IL-TEAM002 | 开发必须并行 | 串行 = 低效 |
| IL-TEAM003 | 监督必须持续 | 不监控 = 放任 |
| IL-TEAM004 | 结果必须用户确认 | 自作主张 = 越界 |
| IL-TEAM005 | 禁止单线程退化 | 主 Agent 自己干活 = 违背并行原则 |

## 技术事实

铁律生效的方式——理解这些事实，让 AI 自己推理为何不能绕过：

1. **铁律在 SessionStart Hook 注入** — 每次会话启动，铁律上下文直接进入系统提示，不是可选参考
2. **PreToolUse Hook 拦截** — Write/Edit 操作前自动检查 IL001（版本目录）和 IL005（敏感配置），返回 exit 1 即阻断
3. **Stop Hook 检测** — 每轮结束自动分析完成声明，无验证证据 = IL003 触发
4. **绕过话术会被识别** — "简单修复"、"就这一次"等关键词在 laziness-detect hook 中捕获

绕过铁律的唯一结果是：hook 阻断 → 铁律日志记录 → 学习分析 → 规则收紧。

## 绕过检测

### 绕过话术模式

| 话术类型 | 示例 | 为什么是绕过 | 反驳 |
|---------|------|-------------|------|
| 简化论 | "简单修复"、"小改动" | 简化 ≠ 简化验证 | 简化的是操作，不是验证标准 |
| 例外论 | "就这一次"、"特殊情况" | 铁律的效力在于一致性 | 铁律无例外，否则就不是铁律 |
| 效率论 | "这样更快"、"省时间" | 跳过步骤的时间会花在回滚上 | 短期快 = 长期慢 |
| 权威论 | "用户要求的"、"紧急" | 紧急也需要流程保障 | 激活 overdrive 超频模式 |
| 推卸论 | "建议用户手动"、"超出范围" | 这是职责范围 | 请继续执行 |

### LP007: Team 阶段主 Agent 代劳

当子 Agent 未产出时，主 Agent 自己接手做同样事情 → 严重违规。

**正确行为**：重新 spawn Agent 或追加冗余 Agent，主 Agent 永远不接手。

### 触发词检测

检测到以下关键词时自动激活铁律检查：
- 中文：简单修复、跳过测试、就这一次、快速处理、特殊情况
- 英文：simple fix、skip test、just this once、quick fix、special case

## 深度防御框架（Defense-in-Depth）

### 4 层验证框架

铁律的执行不仅是阻断，还有纵深防御。每层验证捕获不同类型的错误，层层叠加确保安全。

| 层 | 名称 | 职责 | 铁律对应 |
|---|------|------|---------|
| **Layer 1** | 入口验证 | 拒绝明显无效输入 | IL001（版本目录格式检查） |
| **Layer 2** | 业务逻辑验证 | 确保操作对上下文有意义 | IL002（扫描数据检查）、IL005（敏感配置） |
| **Layer 3** | 环境守卫 | 防止特定上下文中的危险操作 | IL004（版本变更确认）、IL-TEAM004（用户确认） |
| **Layer 4** | 调试仪表 | 捕获法医证据 | IL003（完成声明验证） |

**Layer 1: 入口验证**
- 检查输入是否为空
- 验证文件格式是否正确
- 确认必需参数存在

**Layer 2: 业务逻辑验证**
- 检查依赖关系是否满足
- 验证权限和访问控制
- 确认数据一致性

**Layer 3: 环境守卫**
- 防止生产环境危险操作
- 检查未提交修改
- 验证环境配置

**Layer 4: 调试仪表**
- 记录操作日志
- 捕获堆栈跟踪
- 保存环境变量

### 深度防御与 Gate 状态机的集成

| Gate 类型 | 对应防御层 | 说明 |
|-----------|-----------|------|
| gate-quality-iron-law | Layer 1 + 2 | 铁律违规检查 + 上下文验证 |
| gate-quality-tests | Layer 2 + 4 | 测试通过 + 测试日志记录 |
| gate-quality-format | Layer 1 | 格式基础检查 |
| Stage Gates (W01-W12) | Layer 3 | 阶段条件 + 环境守卫 |

### 使用场景

| 场景 | 使用层 |
|------|--------|
| 文件写入 | Layer 1 + 2 + 4（版本目录 + 内容验证 + 日志记录） |
| 配置变更 | Layer 1 + 2 + 3（格式 + 权限 + 环境守卫） |
| 阶段切换 | Layer 3（阶段条件验证 + Gate 检查） |
| 代码提交 | Layer 1 + 2 + 3 + 4（全层验证） |

## 超频模式 (Overdrive)

当用户激活超频模式时，铁律执行方式调整：

| 铁律 | 超频模式调整 |
|------|-------------|
| IL001 | **保持** — 输出必须在版本目录 |
| IL002 | **跳过** — 无需前置扫描，直接基于已有上下文 |
| IL003 | **保持** — 解决后必须附带验证证据 |
| IL004 | **保持** — 版本号不可擅自修改 |
| IL005 | **保持** — 敏感操作仍需用户确认 |
| IL-TEAM001~005 | **加强** — 强制多 Agent 并行，禁止单线程退化 |

**核心原则**：超频模式跳过的是"前置流程"，不是"铁律验证"。所有底线铁律仍然生效。

详见 `skills/overdrive/SKILL.md`。

## 用户自定义铁律

用户可以在 `~/.claude/harness/iron-laws.yaml` 中添加项目专属铁律：

```yaml
custom_iron_laws:
  - id: IL-C001
    rule: "NO DATABASE CHANGES WITHOUT BACKUP"
    severity: critical
    triggers:
      - pattern: "ALTER TABLE|DROP TABLE"
        action: block
```

铁律 ID 命名规范：
- `IL001-IL099` — 核心铁律（保留）
- `IL-C001-IL-C099` — 用户自定义
- `IL-J001-IL-J099` — Java 栈特定
- `IL-N001-IL-N099` — Node 栈特定

## 执行记录

铁律触发后，追加记录到日志：

**bash 命令（可直接执行）：**
```bash
# 追加铁律触发记录
echo '{"iron_law":"IL001","context":"文档输出无版本目录","action":"block","timestamp":"'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"}' | \
  jq -s '.' >> ~/.claude/harness/iron-law-log.json 2>/dev/null || true
```

所有触发记录在 `~/.claude/harness/iron-law-log.json`，格式：
```json
[
  {"iron_law":"IL001","context":"...","action":"block","timestamp":"..."}
]
```

## 禁止理由表 (Bypass Patterns)

每条铁律都有对应的禁止理由表，用于识别和反驳常见的绕过话术：

### IL001 禁止理由

| 借口 | 反驳 |
|------|------|
| "就这一次，快速处理" | IL001 没有例外。如果紧急，使用 /overdrive 激活紧急模式 |
| "这只是临时文件" | 临时文件也应遵循版本目录。可使用 output/v1.3.1/temp/ |
| "用户要求跳过" | IL001 不可跳过。用户权限需通过 /iron-law-enforcer override 确认 |

### IL002 禁止理由

| 借口 | 反驳 |
|------|------|
| "我已经了解项目结构" | 主观了解不够，需要扫描数据确认。使用 /project-scanner 快速扫描 |

### IL-TEAM005 禁止理由

| 借口 | 反驳 |
|------|------|
| "这个任务太简单，不需要多 Agent" | 简单任务也需要并行验证。使用 /agent-team-orchestrator 分配最少 2 个 Agent |
| "子 Agent 不产出，我直接做" | LP007 检测已触发。正确做法：重新 spawn / 追加冗余 Agent / 只汇总结果 |
