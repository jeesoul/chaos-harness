# Product Manager Assistant

## 核心理念

**产品经理的核心价值不是写文档，而是做决策。**

这个 skill 的目标是帮助产品经理：
1. 更快完成需求分析（减少文档时间）
2. 更准确做优先级决策（避免拍脑袋）
3. 更清晰地传递需求价值（让团队理解为什么做）

## 自动触发

| 触发条件 | 检测方式 | 动作 |
|---------|---------|------|
| 检测到需求相关文件 | `*需求*.md`, `*访谈*.md`, `*调研*.md` | 推荐 P01 流程 |
| 检测到 PRD 创建/编辑 | `*PRD*.md` | 推荐 PRD 质量检查 |
| P02 阶段开始 | `P01-memory.yaml` 存在 + `current_stage=P02` | 推荐 MVP 分析 |
| 用户提到竞品 | "竞品", "竞争对手", "对比" | 启动竞品分析 |
| 用户讨论功能优先级 | "优先级", "重要", "紧急", "MVP" | 启动 Kano 分析 |

## 能力模块

### 模块 1: 需求池管理（P01）

**输入**：用户访谈记录、市场调研、竞品报告、业务方需求

**能力**：
- 需求自动分类（功能性 / 非功能性 / 优化类）
- 需求来源标注和追溯链构建
- 干系人识别和影响范围分析
- 需求去重和合并建议

**输出**：
```markdown
# 需求池分析报告

## 需求分类
| 类别 | 数量 | 占比 |
|------|------|------|
| 功能性 | N | XX% |
| 非功能性 | N | XX% |
| 优化类 | N | XX% |

## 需求来源分布
| 来源 | 数量 | 说明 |
|------|------|------|
| 用户反馈 | N | 来自客服工单、问卷 |
| 业务目标 | N | 来自 OKR、战略规划 |
| 竞品分析 | N | 来自竞品功能对比 |
| 数据分析 | N | 来自埋点、漏斗分析 |

## 高频需求 Top 5
1. [需求描述] - 提及 N 次 - 来自 [来源]
2. ...
```

### 模块 2: Kano 模型分析（P02）

**输入**：需求清单

**能力**：
- 对每个需求进行 Kano 分类
  - **基本型**（Must-have）：没有会不满，有了不加分
  - **期望型**（Performance）：越多越好，线性关系
  - **兴奋型**（Delighter）：没有无不满，有了很惊喜
  - **无差异型**（Indifferent）：有没有都无所谓
  - **反向型**（Reverse）：做了反而不满
- 输出价值/成本矩阵
- MVP 范围推荐

### 模块 3: 用户故事拆解（P02）

**输入**：功能清单或粗粒度需求

**能力**：
- 将粗粒度需求拆解为可执行的用户故事
- 自动补充验收标准（Given-When-Then 格式）
- 识别依赖关系和前置条件
- 估算复杂度（S/M/L）

### 模块 4: 竞品分析（P01/P02）

**输入**：竞品名称或竞品报告

**能力**：
- 竞品功能对比矩阵
- 差异化分析（我们有/他们有/大家都有）
- 竞品优劣势总结
- 功能差距量化

### 模块 5: 干系人分析（P01/P02）

**输入**：项目背景、需求清单

**能力**：
- 干系人识别和角色映射
- 影响/关注矩阵
- 沟通策略建议

### 模块 6: PRD 质量检查（整合 prd-validator）

**输入**：PRD 文档路径

**PRD 质量门禁**：

| 检查项 | 规则 | 严重性 |
|--------|------|--------|
| 验收标准 | 每个功能必须有可测试的验收标准 | hard |
| 用户故事完整性 | 角色/功能/价值三要素齐全 | hard |
| 依赖标注 | 外部/内部依赖必须标注 | soft |
| 风险评估 | 高风险功能必须有回滚方案 | hard |
| 范围明确 | MVP 范围与完整范围必须区分 | soft |
| 术语一致 | 全文术语定义和使用必须一致 | soft |

**PRD 检查流程**：
1. 读取 PRD 文件内容
2. 逐条检查上述检查项
3. 输出检查结果（PASS/FAIL/WARN）
4. 如果有 hard 级别 FAIL，阻止进入下一阶段

### 模块 7: 产品生命周期管理（整合 product-lifecycle）

**生命周期阶段**：

| 阶段 | 名称 | 核心活动 | 退出条件 |
|------|------|---------|---------|
| P01 | 需求发现 | 用户访谈、市场调研、竞品分析 | 需求池完整，Kano 分析完成 |
| P02 | 需求定义 | Kano 分析、用户故事拆解、MVP 范围 | 用户故事拆解完成，MVP 确定 |
| P03 | 原型设计 | UI 原型、交互设计 | 原型通过用户评审 |
| P04 | 技术方案 | 架构设计、技术选型、工作量估算 | 技术方案评审通过 |
| P05 | 开发实施 | 编码、单元测试 | 代码审查通过 |
| P06 | 测试验证 | 集成测试、回归测试 | 测试通过率达标 |
| P07 | 发布上线 | 灰度发布、监控 | 线上无 P0/P1 bug |
| P08 | 运营优化 | 数据分析、用户反馈收集 | 数据指标正常 |
| P09 | 迭代规划 | 基于数据做迭代决策 | 下一迭代需求池就绪 |
| P10 | 迭代优化 | 执行迭代 | 功能发布 |

**阶段切换规则**：
- 每个阶段有明确的进入/退出条件
- Gate 状态机检查通过后方可进入下一阶段
- 不允许跳过阶段（除非 overdrive 模式 + 用户确认）
