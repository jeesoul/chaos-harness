# Product Manager Assistant

## 核心理念

**产品经理的核心价值不是写文档，而是做决策。**

这个 skill 的目标是帮助产品经理：
1. 更快完成需求分析（减少文档时间）
2. 更准确做优先级决策（避免拍脑袋）
3. 更清晰地传递需求价值（让团队理解为什么做）

## 自动触发

| 触发条件 | 检测方式 | 动作 |
|---------|---------|------|
| 检测到需求相关文件 | `*需求*.md`, `*访谈*.md`, `*调研*.md` | 推荐 P01 流程 |
| 检测到 PRD 创建/编辑 | `*PRD*.md` | 推荐 prd-validator |
| P02 阶段开始 | `P01-memory.yaml` 存在 + `current_stage=P02` | 推荐 MVP 分析 |
| 用户提到竞品 | "竞品", "竞争对手", "对比" | 启动竞品分析 |
| 用户讨论功能优先级 | "优先级", "重要", "紧急", "MVP" | 启动 Kano 分析 |

## 能力模块

### 模块 1: 需求池管理（P01）

**输入**：用户访谈记录、市场调研、竞品报告、业务方需求

**能力**：
- 需求自动分类（功能性 / 非功能性 / 优化类）
- 需求来源标注和追溯链构建
- 干系人识别和影响范围分析
- 需求去重和合并建议

**输出**：
```markdown
# 需求池分析报告

## 需求分类
| 类别 | 数量 | 占比 |
|------|------|------|
| 功能性 | N | XX% |
| 非功能性 | N | XX% |
| 优化类 | N | XX% |

## 需求来源分布
| 来源 | 数量 | 说明 |
|------|------|------|
| 用户反馈 | N | 来自客服工单、问卷 |
| 业务目标 | N | 来自 OKR、战略规划 |
| 竞品分析 | N | 来自竞品功能对比 |
| 数据分析 | N | 来自埋点、漏斗分析 |

## 高频需求 Top 5
1. [需求描述] - 提及 N 次 - 来自 [来源]
2. ...
```

### 模块 2: Kano 模型分析（P02）

**输入**：需求清单

**能力**：
- 对每个需求进行 Kano 分类
  - **基本型**（Must-have）：没有会不满，有了不加分
  - **期望型**（Performance）：越多越好，线性关系
  - **兴奋型**（Delighter）：没有无不满，有了很惊喜
  - **无差异型**（Indifferent）：有没有都无所谓
  - **反向型**（Reverse）：做了反而不满
- 输出价值/成本矩阵
- MVP 范围推荐

**输出**：
```markdown
# Kano 分析报告

## 需求分类
| 需求 | Kano 类型 | 价值 | 成本 | 建议 |
|------|----------|------|------|------|
| 用户登录 | 基本型 | 高 | 低 | ✅ 必做 |
| 搜索功能 | 期望型 | 高 | 中 | ✅ MVP |
| 暗黑模式 | 兴奋型 | 中 | 低 | ⚠️ 可延后 |

## MVP 范围推荐
### 必须包含（基本型 + 高价值期望型）
- [功能列表]

### 可以延后（兴奋型 + 低价值功能）
- [功能列表]

### 价值/成本矩阵
高价值/低成本 → 优先做
高价值/高成本 → 重点规划
低价值/低成本 → 快速做
低价值/高成本 → 砍掉
```

### 模块 3: 用户故事拆解（P02）

**输入**：功能清单或粗粒度需求

**能力**：
- 将粗粒度需求拆解为可执行的用户故事
- 自动补充验收标准（Given-When-Then 格式）
- 识别依赖关系和前置条件
- 估算复杂度（S/M/L）

**输出**：
```markdown
# 用户故事拆解

## F001: [功能名称]

### 主用户故事
作为 [角色]
我希望 [功能]
以便 [价值]

### 拆分用户故事
| ID | 故事 | 复杂度 | 依赖 |
|----|------|--------|------|
| F001-1 | [子故事1] | S | 无 |
| F001-2 | [子故事2] | M | F001-1 |

### 验收标准（Given-When-Then）
**AC-1: 正常流程**
- Given 用户已登录
- When 点击 XX 按钮
- Then 页面跳转到 XX，并展示 XX 内容

**AC-2: 异常流程**
- Given 网络异常
- When 发起请求
- Then 显示错误提示 "网络异常，请稍后重试"

### 前置条件
- [前置条件列表]

### 依赖项
- [外部依赖/内部依赖]
```

### 模块 4: 竞品分析（P01/P02）

**输入**：竞品名称或竞品报告

**能力**：
- 竞品功能对比矩阵
- 差异化分析（我们有/他们有/大家都有）
- 竞品优劣势总结
- 功能差距量化

**输出**：
```markdown
# 竞品分析报告

## 竞品功能对比
| 功能 | 我方 | 竞品A | 竞品B | 竞品C |
|------|------|-------|-------|-------|
| 功能1 | ✅ | ✅ | ❌ | ✅ |
| 功能2 | ❌ | ✅ | ✅ | ❌ |

## 差异化分析
### 我方独有
- [功能/优势]

### 竞品有但我方没有（需补齐）
- [功能] - 竞品A/B 具备 - 用户期待度: 高/中/低

## 优劣势总结
| 竞品 | 优势 | 劣势 | 威胁等级 |
|------|------|------|---------|
| 竞品A | ... | ... | 高/中/低 |
```

### 模块 5: 干系人分析（P01/P02）

**输入**：项目背景、需求清单

**能力**：
- 干系人识别和角色映射
- 影响/关注矩阵
- 沟通策略建议

## 快捷命令

| 你说 | 动作 |
|------|------|
| "分析需求池" | 模块 1: 需求池分类和来源分析 |
| "Kano 分析" | 模块 2: 需求 Kano 分类和 MVP 推荐 |
| "拆解用户故事" | 模块 3: 粗粒度需求拆解为用户故事 |
| "竞品分析" | 模块 4: 竞品功能对比和差异化 |
| "干系人分析" | 模块 5: 识别干系人和沟通策略 |
| "帮我写 PRD" | 使用模板创建 PRD + 自动检查 |
| "检查 PRD" | 调用 prd-validator |
