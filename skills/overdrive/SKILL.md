# 超频模式 (Overdrive)

## 核心哲学

**超频模式不是正常流程的加速版，而是完全不同的作战状态。**

正常流程是"规划→执行→验证→评审"的线性过程。超频模式是"定位→决策→解决→验证"的并行闪电战——所有 Agent 同时从不同方向切入，最短时间收敛到解决方案并执行。

**这不是绕过铁律，而是在铁律保护下用最高效的方式解决问题。**

## 适用范围

超频模式不限于 bug 修复，适用于**任何角色**的紧急场景：

| 角色 | 紧急场景示例 |
|------|-------------|
| 产品经理 | 需求优先级突变、MVP 范围紧急调整、竞品突发上线 |
| 架构师 | 架构文档重大修改、技术方案紧急变更 |
| 开发 | 线上 bug、热修复、P0 缺陷、生产事故 |
| 测试 | 测试用例紧急调整、E2E 优先级提升、回归范围扩大 |
| 运维 | 发布回滚、监控告警触发、基础设施异常 |
| 售前 | 竞品对比紧急更新、客户案例紧急补充 |

## 激活条件

满足以下任一条件时自动进入超频模式：

| 触发条件 | 检测方式 |
|---------|---------|
| 用户说 "紧急"/"超频"/"overdrive"/"优先处理" | 自然语言触发词 |
| 用户说 "立刻解决"/"马上"/"热修复" | 紧急语义触发词 |
| 用户说 "线上问题"/"生产事故"/"P0" | 生产环境触发词 |
| auto-context 检测到用户情绪紧迫（短促指令、连续催促） | 语言模式检测 |

## 超频模式规则

### 模式切换

进入超频模式后，立即输出模式切换通知：

```
<HARNESS_OVERDRIVE>
⚡ 超频模式已激活
- 优先级: P0 (最高)
- 状态: 所有 Agent 全速运行
- 目标: 最短时间最优解决
</HARNESS_OVERDRIVE>
```

### 铁律适配

超频模式下，铁律执行方式调整但不取消：

| 铁律 | 正常模式 | 超频模式 |
|------|---------|---------|
| IL001 版本目录 | 必须遵守 | **遵守** — 输出仍需在版本目录 |
| IL002 扫描数据 | 必须先扫描 | **跳过前置扫描** — 直接基于已有上下文 |
| IL003 完成验证 | 必须验证 | **遵守** — 解决后必须验证 |
| IL004 版本变更 | 需用户确认 | **遵守** — 版本号不可擅自改 |
| IL005 敏感配置 | 需审批 | **遵守** — 敏感操作仍需确认 |
| IL-TEAM001 多 Agent | 评审时启动 | **立即启动** — 所有 Agent 并行 |
| IL-TEAM002 并行开发 | 可拆分时并行 | **强制并行** — 能拆必须拆 |
| IL-TEAM003 持续监督 | 全程监控 | **加速监督** — 空闲超时减半，1/2/3/5 分钟 |
| IL-TEAM004 用户确认 | 汇总后确认 | **快速确认** — 精简报告，只输出核心结论 |
| IL-TEAM005 禁止单线程 | 必须并行 | **底线不降级** — 绝不单线程 |

### 跳过规则

超频模式下允许跳过以下正常流程步骤（非铁律）：

| 可跳过的步骤 | 原因 |
|-------------|------|
| 项目扫描 | 直接基于已有知识 |
| 完整 Harness 生成 | 使用最小约束集 |
| 详细文档编写 | 精简输出修复报告 |
| 完整评审流程 | 快速验证替代 |

### 绝对禁止

以下行为在任何情况下都不可做：

- ❌ 单线程串行执行（IL-TEAM005 底线）
- ❌ 声称"先看看"但不 spawn Agent
- ❌ 跳过根因分析直接改（LP002）
- ❌ 声称完成但无验证（LP001）
- ❌ 以"紧急"为理由跳过铁律（iron-law-enforcer 检测）

## 大模型效率指令

超频模式激活后，以下效率指令自动生效：

### 输出约束

| 规则 | 说明 |
|------|------|
| **零铺垫** | 不说"好的让我来看看"、"我来分析一下"，直接输出结果 |
| **不解释** | 不说"我这样做是因为..."，直接给出方案 |
| **不铺垫** | 不说"首先我们需要..."，直接 spawn Agent |
| **摘要优先** | 每个 Agent 只输出核心发现，不输出推理过程 |
| **不说废话** | 不总结、不回顾、不铺垫 |

### Agent 调度约束

| 规则 | 说明 |
|------|------|
| **立即分治** | 识别到可拆分任务后 30 秒内必须 spawn Agent |
| **最大并行** | 能拆必须拆，绝不串行 |
| **快速拍板** | 分歧时主 Agent 1 分钟内决策，不展开讨论 |
| **不反复协商** | 每个 Agent 只输出一次结果，不多轮讨论 |
| **主 Agent 不干活** | 主 Agent 只做协调和决策，不做具体分析 |

### Context 优化

| 规则 | 说明 |
|------|------|
| **最小上下文** | 子 Agent 只携带必要信息，不携带完整背景 |
| **不重复加载** | 已加载的 skill 不再重复加载 |
| **跳过无关铁律** | 非底线铁律不在超频模式下反复确认 |

### 超时自动收敛

| 阶段 | 超时 | 自动动作 |
|------|------|---------|
| 并行定位 | 3 分钟 | 汇总已有结果，直接进入决策 |
| 收敛决策 | 1 分钟 | 主 Agent 直接判断 |

## 核心流程

### 第一阶段：并行定位（3+ Agent 同时启动）

根据问题类型动态分配 Agent 角色：

| 问题类型 | Agent 配置 | 职责 |
|---------|-----------|------|
| **代码缺陷** | diagnoser, context-reader, hypothesis-generator | 分析错误、读代码、生成假设 |
| **需求变更** | product_analyst, impact_reader, scope_adjuster | 分析变更、读影响、调整范围 |
| **架构修改** | arch_reader, impact_analyst, alternative_explorer | 读现有架构、分析影响、探索替代方案 |
| **测试调整** | test_analyst, coverage_reader, case_adjuster | 分析测试需求、读覆盖率、调整用例 |
| **运维事故** | log_analyst, infra_reader, fix_generator | 分析日志、读基础设施、生成修复方案 |

每个 Agent 独立工作，不等待其他人结果。主 Agent 只做协调，不做具体分析。

### 第二阶段：收敛决策

1. 汇总各 Agent 分析结果
2. 识别共识 → 高置信度，直接进入解决
3. 识别分歧 → 主 Agent 快速判断（1 分钟内），不展开讨论
4. 输出最优方案决策

### 第三阶段：并行执行

| 执行类型 | 并行 Agent |
|---------|-----------|
| 代码修改 | fixer（修改代码）+ verifier（铁律验证） |
| 文档修改 | writer（修改文档）+ reviewer（质量验证） |
| 配置变更 | fixer（修改配置）+ security_reviewer（安全审查） |
| 测试调整 | case_writer（编写用例）+ coverage_checker（覆盖率验证） |

### 第四阶段：验证输出

1. 执行验证（测试/截图/日志）
2. 输出精简修复报告
3. 经验沉淀到 emergency-patterns

## 修复报告格式

```markdown

| 字段 | 内容 |
|------|------|
| 问题 | {一句话描述} |

```

## 超时控制

| 阶段 | 超时 | 动作 |
|------|------|------|
| 并行定位 | 3 分钟 | 汇总已有结果，直接进入决策 |
| 收敛决策 | 1 分钟 | 主 Agent 直接判断 |
| 并行执行 | 8 分钟 | 输出部分结果，记录未完成项 |
| 验证输出 | 2 分钟 | 手动验证标记 |

总目标：**14 分钟内完成定位→决策→执行→验证全流程**。超时向用户报告。

## 大模型效率保障

超频模式下主 Agent 必须遵守以下效率约束：

### Prompt 精简

- 子 Agent prompt 只包含：目标 + 产出标准 + 约束
- 不嵌入完整背景，只传递必要上下文
- 不说废话，不解释，不铺垫

### 快速收敛

- 不展开多轮讨论，每个 Agent 只输出一次结果
- 分歧时主 Agent 1 分钟内拍板，不反复协商
- 超过 2 次分析仍无结论时，直接输出最可能的方案并标记置信度

### 跳过优化

- 不写详细实现计划，直接动手
- 不做完整的风险评估，只标注高风险点
- 不生成完整的变更日志，只记录核心修改

## 与新系统集成

### 与本能系统的紧急预警

超频模式激活时，自动检查最近的铁律违规记录：

```bash
/instinct-system status --type=iron_law_violation --min-confidence=0.5
```

如果存在高置信度违规模式 → 在定位阶段直接规避已知风险。

### 与评测系统的快速验证

超频模式下的验证流程精简为：

1. 执行相关评测（如存在定义）
2. 只检查 pass@1（首次通过率）
3. 如果失败 → 标记风险，不阻塞输出

```bash
# 快速检查评测状态
/eval-harness report <eval-id>
# 只关注 pass@1 是否通过
```

### 与迭代检索的快速定位

超频模式第一阶段并行定位时，每个 Agent 使用快速检索：

```
检索 Agent 快速检索策略：
1. 直接搜索最高优先级关键词
2. 只评估前 10 个结果
3. 输出 Top 3 最相关文件
4. 不执行完整 4 阶段循环
```

### 与 Schema 工作流的紧急跳过

超频模式下允许临时跳过 Schema 阶段依赖：

```bash
# 强制转换（不推荐，仅限超频模式）
/workflow-supervisor force-transition --from=X --to=Y --reason="超频模式紧急修复"
```

跳过的阶段必须在修复后补充完成。

### 与战略压缩的上下文优化

超频模式下工具调用计数器自动加速：

| 正常模式阈值 | 超频模式阈值 |
|-------------|-------------|
| 50 次警告 | 25 次警告 |
| 100 次强制压缩 | 50 次强制压缩 |
| 150 次 PreCompact | 75 次 PreCompact |

超频模式下每次 spawn 子 Agent 前自动检查计数器，超过 50 次时传递精简上下文。
