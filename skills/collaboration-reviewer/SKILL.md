# 协作评审系统 (Collaboration Reviewer)

## 核心理念

**设计思路和评审需要多人多视角讨论，避免单人盲点。**
**所有协作插件在 Harness 约束下运行，Harness 是最高优先级。**

## 评审阶段映射

| 工作流阶段 | 评审类型 | 推荐参与者 |
|-----------|---------|-----------|
| W02 需求评审 | 需求合理性 | architect, product_manager, user |
| W04 架构评审 | 架构可行性 | architect, senior_dev, security_expert |
| W09 代码审查 | 代码质量 | code_reviewer, tester, architect |

## 多 Agent 评审流程

### Step 1: 派遣评审 Agent

使用 Agent 工具派遣多个评审者：

```
Agent 1: architect - 架构视角评审
Agent 2: security_expert - 安全视角评审  
Agent 3: senior_dev - 实现可行性评审
```

### Step 2: 并行评审

每个 Agent 独立评审，输出评审报告：

```markdown
## 评审报告 - {Agent角色}

### 评分: {1-10}

### 发现的问题
1. {问题1}
2. {问题2}

### 建议改进
1. {建议1}
2. {建议2}

### 关键决策点
- {决策点}: 建议 {选项}
```

### Step 3: 汇总讨论

收集所有 Agent 报告，找出：
- 共识点（所有 Agent 同意）
- 争议点（Agent 意见分歧）
- 风险点（任一 Agent 标记为高风险）

### Step 4: 输出评审结果

```
┌─────────────────────────────────────────────────────────────┐
│  📋 评审汇总报告                                              │
├─────────────────────────────────────────────────────────────┤
│  参与评审: architect, security_expert, senior_dev            │
│  平均评分: 7.5/10                                            │
├─────────────────────────────────────────────────────────────┤
│  ✅ 共识点 (3)                                               │
│    1. 架构设计整体合理                                       │
│    2. 数据库选型正确                                         │
│    3. API 设计清晰                                          │
├─────────────────────────────────────────────────────────────┤
│  ⚠️ 争议点 (2)                                               │
│    1. 缓存方案: architect建议Redis, senior_dev建议本地缓存   │
│    2. 认证方式: JWT vs Session                               │
├─────────────────────────────────────────────────────────────┤
│  🚨 风险点 (1)                                               │
│    1. security_expert: SQL注入风险未处理                     │
└─────────────────────────────────────────────────────────────┘
```

## 用户确认机制

### 争议点投票

对于争议点，让用户投票决定：

```
┌─────────────────────────────────────────────────────────────┐
│  ⚖️ 需要用户决策                                              │
├─────────────────────────────────────────────────────────────┤
│  争议点: 缓存方案                                            │
│                                                              │
│  [A] Redis (architect 推荐)                                  │
│      - 分布式支持                                            │
│      - 持久化                                                │
│                                                              │
│  [B] 本地缓存 (senior_dev 推荐)                              │
│      - 简单                                                  │
│      - 低延迟                                                │
│                                                              │
│  [C] 两者结合                                                │
│      - 本地缓存 + Redis                                      │
│                                                              │
│  请选择方案 (A/B/C):
└─────────────────────────────────────────────────────────────┘
```

### 最终确认

评审完成后，需要用户确认：

```
┌─────────────────────────────────────────────────────────────┐
│  ✅ 评审完成，请确认                                          │
├─────────────────────────────────────────────────────────────┤
│  评审类型: 架构评审 (W04)                                     │
│  参与者: 3 Agents                                            │
│  最终评分: 8/10                                              │
│  已解决争议: 2                                               │
│  剩余风险: 1 (已标记为待处理)                                 │
├─────────────────────────────────────────────────────────────┤
│  [1] ✅ 确认通过，进入下一阶段                                │
│  [2] 🔄 需要修改后重新评审                                    │
│  [3] ❌ 拒绝，需要重新设计                                    │
└─────────────────────────────────────────────────────────────┘
```

## 铁律检查

### IL003: NO COMPLETION CLAIMS WITHOUT VERIFICATION

评审完成必须：
- 至少 2 个 Agent 参与评审
- 所有风险点已标记处理方案
- 用户明确确认

### 自定义规则: 多视角原则

```
CRITICAL DECISIONS REQUIRE MULTIPLE PERSPECTIVES
```

关键决策必须经过多 Agent 评审，不能单人决定。

## 配置文件

评审配置保存在 `.claude/harness/review-config.yaml`:

```yaml
review:
  min_agents: 2
  required_consensus: 0.6  # 60% 同意即通过
  stages:
    W02:
      agents: [architect, product_manager]
      plugins: [superpowers:brainstorming]
    W04:
      agents: [architect, senior_dev, security_expert]
      plugins: [superpowers:brainstorming]
    W09:
      agents: [code_reviewer, tester]
      plugins: [superpowers:code-reviewer, superpowers:requesting-code-review]
```
